Introduction

DainnUser is a comprehensive .NET/C# class library that provides enterprise-grade user management capabilities for your applications. Built with security, flexibility, and developer experience in mind.

🎯 Purpose: DainnUser eliminates the need to build authentication and user management from scratch. Simply install the NuGet package, configure your preferences, and you're ready to go.

Why DainnUser?

  • Production-Ready: OWASP Top 10 compliant with built-in security best practices
  • Flexible: Enable/disable features via configuration - use only what you need
  • Database Agnostic: Supports SQL Server, PostgreSQL, MySQL, and SQLite
  • Clean Architecture: Well-structured, testable, and maintainable codebase
  • Extensible: Easy to customize and extend for your business logic

Target Audience

.NET developers building web applications, APIs, or microservices that require user authentication and management.

Installation

Prerequisites

  • .NET 8.0 SDK or later
  • Database: SQL Server, PostgreSQL, MySQL, or SQLite
  • SMTP Server: For email functionality (Gmail, SendGrid, or local SMTP for dev)

Install via NuGet

Install the required packages in your ASP.NET Core project:

# Core package (required)
dotnet add package DainnUser.Core

# Infrastructure package (required for database)
dotnet add package DainnUser.Infrastructure

# Application package (required for business logic)
dotnet add package DainnUser.Application

# API package (required for controllers)
dotnet add package DainnUser.Api

# Web package (optional - for Razor/Blazor components)
dotnet add package DainnUser.Web
💡 Tip: For most applications, you'll need Core, Infrastructure, Application, and Api packages. The Web package is only needed if you're using server-side Razor or Blazor components.

Manual Installation (Development)

If you're contributing or want to build from source:

git clone https://github.com/dainn/dainn-user.git
cd DainnUser

Quickstart (Auth Only)

Get up and running with basic authentication in 5 minutes.

Step 1: Create a New ASP.NET Core Web API

dotnet new webapi -n MyApp
cd MyApp

Step 2: Install DainnUser Packages

dotnet add package DainnUser.Core
dotnet add package DainnUser.Infrastructure
dotnet add package DainnUser.Application
dotnet add package DainnUser.Api

Step 3: Configure appsettings.json

{
  "DainnUser": {
    "Database": {
      "Provider": "SQLite",
      "ConnectionString": "Data Source=myapp.db"
    },
    "Jwt": {
      "SecretKey": "my-super-secret-jwt-key-at-least-32-chars",
      "Issuer": "MyApp",
      "Audience": "MyAppUsers",
      "ExpirationMinutes": 60
    },
    "Email": {
      "SmtpHost": "localhost",
      "SmtpPort": 1025,
      "FromEmail": "dev@localhost"
    }
  }
}

Step 4: Update Program.cs

var builder = WebApplication.CreateBuilder(args);

// Add DainnUser services
builder.Services.AddDainnUser(builder.Configuration, options =>
{
    options.EnableSocialLogin = false;
    options.EnableTwoFactor = false;
    options.RequireEmailVerification = true;
});

builder.Services.AddControllers();
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen();

var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI();
}

app.UseHttpsRedirection();

// Add DainnUser middleware
app.UseDainnUser();

app.UseAuthorization();
app.MapControllers();

app.Run();

Step 5: Run Migrations

dotnet ef migrations add InitialCreate
dotnet ef database update

Step 6: Run the Application

dotnet run

Navigate to https://localhost:5001/swagger to see the API documentation and test endpoints.

✅ Success! You now have a working authentication system with registration, login, and email verification.

JWT Authentication

How JWT Works in DainnUser

DainnUser uses JWT (JSON Web Tokens) for stateless authentication:

  1. User logs in with credentials
  2. Server validates credentials and generates JWT access token + refresh token
  3. Client stores tokens (typically in localStorage or httpOnly cookies)
  4. Client includes access token in Authorization header for subsequent requests
  5. When access token expires, client uses refresh token to get new access token

JWT Configuration

{
  "DainnUser": {
    "Jwt": {
      "SecretKey": "your-secret-key-minimum-32-characters-long",
      "Issuer": "YourAppName",
      "Audience": "YourAppUsers",
      "ExpirationMinutes": 60,
      "RefreshTokenExpirationDays": 7
    }
  }
}

Login Flow

// POST /api/auth/login
{
  "email": "user@example.com",
  "password": "SecurePass123!"
}

// Response
{
  "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refreshToken": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "expiresIn": 3600,
  "tokenType": "Bearer"
}

Using JWT in Requests

// Add Authorization header to your HTTP requests
GET /api/users/profile
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

Refresh Token Flow

// POST /api/auth/refresh
{
  "refreshToken": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}

// Response
{
  "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refreshToken": "new-refresh-token-uuid",
  "expiresIn": 3600
}

Protecting Endpoints

using Microsoft.AspNetCore.Authorization;

[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
    // Requires authentication
    [Authorize]
    [HttpGet]
    public IActionResult GetProducts()
    {
        // Only authenticated users can access
        return Ok(products);
    }

    // Requires specific role
    [Authorize(Roles = "Admin")]
    [HttpDelete("{id}")]
    public IActionResult DeleteProduct(int id)
    {
        // Only Admin role can access
        return NoContent();
    }

    // Requires specific permission
    [Authorize(Policy = "CanManageProducts")]
    [HttpPost]
    public IActionResult CreateProduct(Product product)
    {
        // Only users with CanManageProducts permission
        return Created();
    }
}

JWT Security Best Practices

  • Secret Key: Use a strong, random key (minimum 32 characters)
  • HTTPS Only: Always use HTTPS in production
  • Short Expiration: Keep access token expiration short (15-60 minutes)
  • Refresh Token Rotation: DainnUser automatically rotates refresh tokens
  • Token Storage: Store tokens securely (httpOnly cookies preferred over localStorage)
  • Logout: Invalidate refresh tokens on logout

Generic OIDC Provider Setup

DainnUser supports any OpenID Connect (OIDC) compliant provider (Auth0, Okta, Keycloak, IdentityServer, etc.).

Enable Generic OIDC

builder.Services.AddDainnUser(builder.Configuration, options =>
{
    options.EnableGenericOidc = true;
});

Configuration Example

{
  "DainnUser": {
    "EnableGenericOidc": true,
    "OidcProviders": [
      {
        "ProviderId": "auth0",
        "DisplayName": "Auth0",
        "Authority": "https://your-tenant.auth0.com",
        "ClientId": "your-client-id",
        "ClientSecret": "your-client-secret",
        "CallbackPath": "/signin-oidc",
        "Scope": "openid profile email",
        "EmailClaimType": "email",
        "NameClaimType": "name",
        "SubjectClaimType": "sub"
      }
    ]
  }
}

OIDC Configuration Parameters

Parameter Required Description
ProviderId Yes Unique identifier for this provider (e.g., "auth0", "okta")
DisplayName Yes Human-readable name shown to users
Authority Yes Base URL of the OIDC provider - used for discovery endpoint
ClientId Yes OAuth 2.0 client identifier
ClientSecret Yes OAuth 2.0 client secret
CallbackPath No Redirect URI path (default: "/signin-oidc")
Scope No Space-separated scopes (default: "openid profile email")
EmailClaimType No Claim type for email (default: "email")
NameClaimType No Claim type for display name (default: "name")
SubjectClaimType No Claim type for subject identifier (default: "sub")
💡 Note: The Authority property is used for OIDC discovery. DainnUser automatically fetches the issuer and endpoints from {Authority}/.well-known/openid-configuration. You don't need to specify a separate Issuer URL - it's retrieved from the discovery document.

Handling External Login Callback

DainnUser automatically handles the OAuth callback. The flow is:

  1. User clicks "Sign in with Google" button
  2. User is redirected to Google login page
  3. User authorizes your app
  4. Google redirects back to /signin-google
  5. DainnUser processes the callback:
    • Validates the OAuth response
    • Creates or updates user account
    • Issues JWT token
    • Redirects to returnUrl

Account Linking

Users can link multiple social accounts to one DainnUser account:

// Link additional provider to existing account
POST /api/auth/link-external-account?provider=Google

// Unlink provider
DELETE /api/auth/unlink-external-account?provider=Google

// Get linked accounts
GET /api/auth/external-accounts

Security Best Practices

⚠️ Important:
  • Use HTTPS: OAuth requires HTTPS in production
  • Validate State Parameter: DainnUser automatically validates to prevent CSRF
  • Store Secrets Securely: Never commit client secrets to source control
  • Limit Scopes: Only request necessary permissions
  • Handle Errors: Gracefully handle OAuth failures
  • Email Verification: Consider requiring email verification even for social logins

Troubleshooting Social Login

Redirect URI Mismatch

Error: "redirect_uri_mismatch" or similar

Solution: Ensure redirect URI in provider settings exactly matches your application URL, including protocol (https://) and port.

Invalid Client

Error: "invalid_client"

Solution: Verify Client ID and Client Secret are correct and not expired.

User Not Created

Error: User logs in but account not created

Solution: Check that EnableSocialLogin = true in DainnUser options.

        await SendEmailAsync(email, "Verify Your Email", htmlBody);
    }
}

// Register in Program.cs
builder.Services.AddScoped<IEmailService, CustomEmailService>();

Troubleshooting

Common Issues

1. Email Not Sending

Problem: Verification emails are not being sent.

Solutions:
  • Check SMTP configuration in appsettings.json
  • For Gmail: Enable "Less secure app access" or use App Password
  • Check firewall/antivirus blocking SMTP port
  • Use MailHog for local development testing
  • Check application logs for email errors

2. JWT Token Invalid

Problem: Getting "401 Unauthorized" with valid token.

Solutions:
  • Verify JWT SecretKey matches between configuration and token generation
  • Check token expiration time
  • Ensure app.UseDainnUser() is called before app.UseAuthorization()
  • Verify Authorization header format: Bearer {token}
  • Check Issuer and Audience match configuration

3. Database Migration Errors

Problem: Migration fails or database not created.

Solutions:
  • Ensure connection string is correct
  • Check database server is running
  • Verify user has permissions to create database/tables
  • For SQLite: Check file path is writable
  • Run migrations from correct project directory

4. Account Lockout Issues

Problem: Account locked after failed attempts.

Solutions:
  • Wait for lockout duration to expire (default: 15 minutes)
  • Admin can manually unlock via database or API
  • Adjust MaxLoginAttempts and LockoutDurationMinutes in config
  • Disable lockout in development: EnableAccountLockout = false

5. CORS Errors

Problem: Frontend can't call API due to CORS.

Solution: Add CORS policy in Program.cs: 
builder.Services.AddCors(options =>
{
    options.AddPolicy("AllowFrontend", policy =>
    {
        policy.WithOrigins("http://localhost:3000")
              .AllowAnyHeader()
              .AllowAnyMethod()
              .AllowCredentials();
    });
});

app.UseCors("AllowFrontend");
