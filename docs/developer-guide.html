DainnUser v1.0 developer guide

Install a complete user module fast, then replace any layer when your app needs custom behavior.

DainnUser ships default authentication, profile, address, contact, role, session, login history, two-factor, social login, email, JWT, storage, and Razor UI building blocks. The default path is one service registration. The custom path is normal .NET dependency injection.

Target.NET 8 applications
Default setupAddDainnUser()
CustomizationRegister overrides before defaults
UIRazor/Blazor event-driven components

Start Here

If you only want a working module quickly, follow this order. Everything else in the guide is reference material for when you need to customize.

Required

1. Install packages

Install DainnUser.Core, Application, and Infrastructure. Add Web only if you need Razor components.

Required

2. Configure secrets

Set database, JWT, and email provider config. Add social login or reCAPTCHA only if the feature is enabled.

Required

3. Register services

Call AddDainnUser() in Program.cs. Add JWT bearer auth in the host app and keep your custom overrides before the default registration.

Required

4. Run migrations

Apply DainnUserDbContext migrations against the target database before testing login and registration.

Optional

5. Add UI

Use the Razor components if you want prebuilt forms. If you already have UI, skip the Web package and call services directly.

Advanced

6. Custom business layer

Build your own request/response models, validators, and services around DainnUser contracts when the app needs product-specific logic.

This guide is structured so you can stop after the first four steps and still have a working module.

Goal

The library is designed for teams that want a user module quickly without locking the application into a closed implementation. Start with defaults, then override the services that need product-specific behavior: email delivery, token generation, repositories, profile workflows, contact verification, storage, or UI event handling.

CoreEntities, interfaces, DTOs, options, exceptions, and contracts.
ApplicationBusiness services, validation, and default use-case workflows.
InfrastructureEF Core, repositories, JWT, email providers, storage, and middleware.
WebReusable Razor components with callback-based submit behavior.
Register custom services before calling AddDainnUser(). Default registrations use TryAdd* semantics so your implementation remains active.

What DainnUser Includes

DainnUser is a user-management library, not a full application shell. It gives developers the identity domain, default business workflows, persistence adapters, security utilities, and optional Razor UI components needed to build a user module quickly.

Authentication

Register, login, refresh token, logout, password reset, password change, email verification, and account lockout.

Authorization

Roles, permissions constants, policy names, user-role assignment, role lookup, and admin user-management contracts.

User Profile

Profile DTOs, profile service, update profile flow, avatar storage hooks, and profile controller examples.

Contacts

Email/phone contact records, primary contact handling, verification code flow, verification attempts, and token persistence.

Addresses

User address models, repository contracts, validators, default service behavior, and API controller support.

Sessions

Session creation, session metadata, active-session limits, session revoke, and login-history tracking.

Two-Factor Auth

2FA setup, code verification, backup codes, remember-device tokens, and completion of pending 2FA login.

Social Login

Google, Facebook, Microsoft, and GitHub option fields plus default service integration points for external provider login.

Security

JWT token service, refresh-token rotation, password hashing, reCAPTCHA hooks, rate-limiting middleware, and security exceptions.

Core Contracts

The Core package contains the stable contracts developers build against. These are the primary extension points when implementing custom business behavior.

Authentication IAuthenticationService, IJwtTokenService, ITwoFactorService Wrapping or replacing auth/token/two-factor services.
User data IUserRepository, IRoleRepository, IUnitOfWork Providing custom repositories or unit of work backed by existing app persistence.

Default Implementations

Package Default implementation Notes
DainnUser.Application Authentication, profile, role, session, login history, social login, reCAPTCHA, address, contact, activity services. Business workflows and validators. Depends on Core contracts, not Infrastructure.
DainnUser.Infrastructure EF Core DbContext, repositories, Unit of Work, JWT token service, email providers, storage providers, rate limiter. Can be used as the default full-stack adapter or partially replaced through DI.
DainnUser.OpenIddict Optional OpenID Connect provider infrastructure, OpenIddict server registration, EF Core stores, and client seeding. Use only when your app must act as an OIDC/OAuth authorization server for other clients.

Quickstart: Password Login and Register

This is the smallest useful Web API setup a developer can copy. It gives register, login, refresh-token, authenticated endpoints, EF Core persistence, JWT bearer validation, and default DainnUser services.

1. Install Packages

Terminal
dotnet add package DainnUser.Core
dotnet add package DainnUser.Application
dotnet add package DainnUser.Infrastructure

2. Minimal appsettings.json

appsettings.json
{
  "DainnUser": {
    "Database": {
      "Provider": "SqlServer",
      "ConnectionString": "Server=localhost;Database=MyAppUsers;Trusted_Connection=True;TrustServerCertificate=True"
    },
    "Jwt": {
      "Secret": "replace-with-a-strong-secret-at-least-32-bytes-long",
      "Issuer": "https://localhost:5001",
      "Audience": "https://localhost:5001"
    },
    "Email": {
      "Provider": "Smtp",
      "FromEmail": "noreply@your-app.example",
      "FromName": "Your App",
      "SmtpHost": "smtp.example.com",
      "SmtpPort": 587,
      "EnableSsl": true
    }
  }
}

3. Minimal Program.cs

Program.cs
using System.IdentityModel.Tokens.Jwt;
using System.Text;
using DainnUser.Core.Authorization;
using DainnUser.Infrastructure;
using DainnUser.Infrastructure.Configuration;
using Microsoft.AspNetCore.Authentication.JwtBearer;
using Microsoft.IdentityModel.Tokens;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();
builder.Services.AddEndpointsApiExplorer();

builder.Services.AddDainnUser(builder.Configuration, options =>
{
    options.RequireEmailVerification = false;
    options.EnableTwoFactor = false;
    options.EnableAccountLockout = true;
    options.EnableSessionManagement = true;
    options.EnableActivityLogging = true;
    options.EnableRateLimiting = false;
});

var jwt = builder.Configuration.GetSection("DainnUser:Jwt").Get<JwtOptions>()!;
var signingKey = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(jwt.Secret));
JwtSecurityTokenHandler.DefaultInboundClaimTypeMap.Clear();

builder.Services
    .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.MapInboundClaims = false;
        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer = jwt.ValidateIssuer,
            ValidateAudience = jwt.ValidateAudience,
            ValidateIssuerSigningKey = true,
            ValidateLifetime = true,
            ValidIssuer = jwt.Issuer,
            ValidAudience = jwt.Audience,
            IssuerSigningKey = signingKey,
            ClockSkew = TimeSpan.FromSeconds(jwt.ClockSkewSeconds)
        };
    });

builder.Services.AddAuthorization();

var app = builder.Build();

app.UseHttpsRedirection();
app.UseAuthentication();
app.UseAuthorization();
app.UseDainnUser();

app.MapControllers();

app.Run();

4. Minimal Auth Controller

Use this controller if the app does not consume the provided API controllers directly or wants its own route/response shape.

AccountController.cs
using DainnUser.Application.DTOs.Authentication;
using DainnUser.Core.Interfaces.Services;
using Microsoft.AspNetCore.Mvc;

[ApiController]
[Route("api/account")]
public sealed class AccountController : ControllerBase
{
    private readonly IAuthenticationService _auth;

    public AccountController(IAuthenticationService auth)
    {
        _auth = auth;
    }

    [HttpPost("register")]
    public async Task<IActionResult> Register(RegisterDto request, CancellationToken ct)
    {
        var userId = await _auth.RegisterAsync(
            request.Email,
            request.Username,
            request.Password,
            ct);

        return Created($"/api/users/{userId}", new { userId });
    }

    [HttpPost("login")]
    public async Task<IActionResult> Login(LoginDto request, CancellationToken ct)
    {
        var result = await _auth.LoginAsync(
            request.Email,
            request.Password,
            HttpContext.Connection.RemoteIpAddress?.ToString(),
            Request.Headers.UserAgent.ToString(),
            request.RememberDeviceToken,
            ct);

        return Ok(result);
    }
}

Which Features Need Appsettings?

Some features are controlled only by DainnUserOptions, while infrastructure-backed features also need appsettings.json. The default AddDainnUser() path validates database, email, and JWT unless you register custom services before calling it.

Feature Needs appsettings? Required keys Related DainnUserOptions
Password login Yes DainnUser:Database, DainnUser:Jwt, and email config when default email service is used. RequireEmailVerification, EnableAccountLockout, EnableSessionManagement, JwtExpirationMinutes
Register user Yes Database. Email config is required if verification emails are sent by default services. RequireEmailVerification, EnableActivityLogging
Password reset Yes Email provider and database token persistence. PasswordResetTokenExpirationHours
JWT access/refresh tokens Yes DainnUser:Jwt:Secret, Issuer, Audience. JwtExpirationMinutes, RefreshTokenExpirationDays
Email verification Yes DainnUser:Email for SMTP, SendGrid, or AWS SES. RequireEmailVerification
Two-factor authentication No for core flow; yes if delivery is customized Database for user secret/tokens. Email/SMS config depends on the delivery implementation. EnableTwoFactor, TwoFactorSetupExpirationMinutes, TwoFactorRememberDeviceDays
Social login: Google, Facebook, Microsoft, GitHub Recommended Provider client IDs/secrets. You may store them under Authentication:{Provider} and map into options. EnableSocialLogin, provider client IDs/secrets/callback paths.
Google reCAPTCHA Recommended Site key and secret key. You may store them under Google:Recaptcha and map into options. RecaptchaEnabled, RecaptchaVersion, RecaptchaSiteKey, RecaptchaSecretKey, RecaptchaMinimumScore
Rate limiting Yes for rules DainnUser:RateLimiting:Enabled, Rules, WhitelistIps. EnableRateLimiting, RateLimitRequestsPerMinute
Avatar/file storage Yes unless custom storage is registered DainnUser:Storage for Local, Azure Blob, or AWS S3. No feature flag. Override IStorageService or IAvatarService for custom behavior.
Profile, contacts, addresses, roles Database only DainnUser:Database. Mostly controlled by service usage; contacts use verification sender if verification is requested.
Activity logging and login history Database only DainnUser:Database. EnableActivityLogging, LoginHistoryRetentionDays

JWT Authentication Setup

DainnUser can generate JWT access tokens through IJwtTokenService. The host application must also validate those tokens for protected endpoints. DainnUser.Api is sample/reference source, so production apps should wire JWT bearer authentication in their own host project.

Program.cs
using System.IdentityModel.Tokens.Jwt;
using System.Text;
using DainnUser.Infrastructure.Configuration;
using Microsoft.AspNetCore.Authentication.JwtBearer;
using Microsoft.IdentityModel.Tokens;

var jwt = builder.Configuration.GetSection("DainnUser:Jwt").Get<JwtOptions>()!;
var signingKey = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(jwt.Secret));
JwtSecurityTokenHandler.DefaultInboundClaimTypeMap.Clear();

builder.Services
    .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.MapInboundClaims = false;
        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer = jwt.ValidateIssuer,
            ValidateAudience = jwt.ValidateAudience,
            ValidateIssuerSigningKey = true,
            ValidateLifetime = true,
            ValidIssuer = jwt.Issuer,
            ValidAudience = jwt.Audience,
            IssuerSigningKey = signingKey,
            ClockSkew = TimeSpan.FromSeconds(jwt.ClockSkewSeconds)
        };
    });

builder.Services.AddAuthorization();

var app = builder.Build();

app.UseAuthentication();
app.UseAuthorization();

Bearer Token Usage

Clients send the returned access token in the standard authorization header.

HTTP
GET /api/profile
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

JWT Config Keys

Key Required Description
DainnUser:Jwt:Secret Yes HMAC signing secret. Must be at least 32 bytes.
DainnUser:Jwt:Issuer Recommended Expected issuer claim. Must match token generation and validation.
DainnUser:Jwt:Audience Recommended Expected audience claim. Must match token generation and validation.
DainnUser:Jwt:ValidateIssuer No Defaults to true. Disable only for controlled development scenarios.
DainnUser:Jwt:ValidateAudience No Defaults to true. Disable only for controlled development scenarios.
DainnUser:Jwt:ClockSkewSeconds No Allowed clock skew during validation. Defaults to 30.

OIDC Provider Module

Use DainnUser.OpenIddict when the host application must become an OpenID Connect provider for other web apps, SPAs, mobile apps, or service clients. Keep this separate from the JSON login API: DainnUser owns user authentication, 2FA, roles, permissions, sessions, and lockout; OpenIddict owns OIDC/OAuth protocol endpoints and client validation.

Integration boundary: Do not move password checks, two-factor checks, or RBAC decisions into OpenIddict handlers. The OIDC module should call DainnUser services, then issue OpenIddict authorization codes/tokens from the resulting authenticated principal.

Install and Register

Reference the optional project/package from the host app, then opt in from Program.cs. Apps that only need DainnUser's built-in JWT login do not need this module.

Program.cs
using DainnUser.Infrastructure;

Authorization Endpoint Contract

The host app still owns the interactive login and consent pages. The /connect/authorize handler should redirect unauthenticated users to the login page, call DainnUser login services, and only sign into OpenIddict after authentication is complete.

1Client sends user to /connect/authorize with response_type=code, client_id, redirect_uri, scope, and PKCE parameters.
2If the browser has no interactive auth cookie, redirect to the host login page and preserve the original authorize request.
3Login page calls IAuthenticationService.LoginAsync. This keeps password validation, email verification, lockout, sessions, and audit history in DainnUser.
4If RequiresTwoFactor is true, redirect to the 2FA page and call CompleteTwoFactorLoginAsync. Do not issue an authorization code yet.
5After full authentication and consent, build a claims principal and return SignIn(principal, OpenIddictServerAspNetCoreDefaults.AuthenticationScheme).

RBAC and Claims

Roles and permissions should be projected into OIDC claims by a small factory owned by the OIDC module or host app. Gate sensitive claims by requested scopes.

Requested scope Suggested claims Notes
openid sub Always required for OIDC. Use the DainnUser user ID as the subject.
email email, email_verified Expose only when the client is allowed to request email data.
profile name, preferred_username Map from username/profile fields. Keep app-specific profile data in app APIs.
roles role Map DainnUser roles. Configure resource APIs to read the same role claim type.
permissions permission Use a custom scope for fine-grained permissions. Do not add permissions by default to every token.

using DainnUser.Core.Interfaces.Services;
using DainnUser.Core.Models.Authentication;

public sealed class AccountWorkflow
{
    private readonly IAuthenticationService _auth;
    private readonly IProfileService _profiles;

    public AccountWorkflow(IAuthenticationService auth, IProfileService profiles)
    {
        _auth = auth;
        _profiles = profiles;
    }

    public async Task<LoginResult> LoginAndPrepareProfileAsync(
        string email,
        string password,
        CancellationToken cancellationToken)
    {
        var result = await _auth.LoginAsync(
            email,
            password,
            ipAddress: null,
            userAgent: null,
            cancellationToken: cancellationToken);

        {
            await _profiles.GetProfileAsync(result.User.Id, cancellationToken);
        }

        return result;
    }
}

Business Extensions: Models, Functions, APIs, Validation

Most product-specific requirements should live in your application layer, not by editing DainnUser internals. Keep DainnUser responsible for identity, security, sessions, roles, contacts, and tokens. Add business fields and workflows in your own module and connect them through UserId.

Preferred approach: extend around DainnUser with your own request/response models, validators, entities, services, and controllers. Use DainnUser interfaces for identity operations.

When to extend outside DainnUser

Need Recommended pattern Why
Add fields like company, department, tax code, customer tier Create a business table with UserId Keeps DainnUser migrations stable and lets your app own business schema.
Add a register workflow with tenant, CRM, default plan, invite code Create your own service that composes IAuthenticationService Uses secure default auth while keeping product workflow custom.
Change validation rules for a product endpoint Create your own request DTO and FluentValidation validator Avoids changing shared DainnUser DTOs for one app's contract.
Return app-specific response shape Create your own response DTO and map from DainnUser result Lets frontend/API contract evolve independently.
Replace core auth behavior Implement and register IAuthenticationService Use only when composition is not enough.

3. Validate the Business Contract

Use FluentValidation for your API request. DainnUser validates its own DTOs; your host app should validate its own custom DTOs before calling DainnUser services.

RegisterCustomerRequestValidator.cs
using FluentValidation;

public sealed class RegisterCustomerRequestValidator : AbstractValidator<RegisterCustomerRequest>
{
    public RegisterCustomerRequestValidator()
    {
        RuleFor(x => x.Email)
            .NotEmpty()
            .EmailAddress()
            .MaximumLength(256);

        RuleFor(x => x.Username)
            .NotEmpty()
            .MinimumLength(3)
            .MaximumLength(64);

        RuleFor(x => x.Password)
            .NotEmpty()
            .MinimumLength(12);

        RuleFor(x => x.ConfirmPassword)
            .Equal(x => x.Password)
            .WithMessage("Passwords do not match.");

        RuleFor(x => x.CompanyName)
            .NotEmpty()
            .MaximumLength(160);

        RuleFor(x => x.TaxCode)
            .MaximumLength(32);
    }
}

4. Create a Business Function Around DainnUser

The custom service owns the business transaction. It calls DainnUser for user creation, then writes your app-specific data and optionally calls other systems.

CustomerRegistrationService.cs
using DainnUser.Core.Interfaces.Services;

public interface ICustomerRegistrationService
{
    Task<RegisterCustomerResponse> RegisterAsync(
        RegisterCustomerRequest request,
        CancellationToken cancellationToken);
}

public sealed class CustomerRegistrationService : ICustomerRegistrationService
{
    private readonly IAuthenticationService _auth;
    private readonly AppDbContext _db;

    public CustomerRegistrationService(IAuthenticationService auth, AppDbContext db)
    {
        _auth = auth;
        _db = db;
    }

    public async Task<RegisterCustomerResponse> RegisterAsync(
        RegisterCustomerRequest request,
        CancellationToken cancellationToken)
    {
        var userId = await _auth.RegisterAsync(
            request.Email,
            request.Username,
            request.Password,
            cancellationToken);

        var profile = new CustomerProfile
        {
            Id = Guid.NewGuid(),
            UserId = userId,
            CompanyName = request.CompanyName,
            TaxCode = request.TaxCode,
            AcceptsMarketing = request.AcceptsMarketing
        };

        _db.CustomerProfiles.Add(profile);
        await _db.SaveChangesAsync(cancellationToken);

        return new RegisterCustomerResponse
        {
            UserId = userId,
            CustomerProfileId = profile.Id,
            Email = request.Email,
            CompanyName = profile.CompanyName,
            RequiresEmailVerification = true
        };
    }
}

5. Expose a Custom API Endpoint

Your API can return your own response format while DainnUser handles identity internals behind the service boundary.

CustomersController.cs
using FluentValidation;
using Microsoft.AspNetCore.Mvc;

[ApiController]
[Route("api/customers")]
public sealed class CustomersController : ControllerBase
{
    private readonly ICustomerRegistrationService _registration;
    private readonly IValidator<RegisterCustomerRequest> _validator;

    public CustomersController(
        ICustomerRegistrationService registration,
        IValidator<RegisterCustomerRequest> validator)
    {
        _registration = registration;
        _validator = validator;
    }

    [HttpPost("register")]
    [ProducesResponseType(typeof(RegisterCustomerResponse), StatusCodes.Status201Created)]
    [ProducesResponseType(StatusCodes.Status400BadRequest)]
    public async Task<IActionResult> Register(
        RegisterCustomerRequest request,
        CancellationToken cancellationToken)
    {
        var validation = await _validator.ValidateAsync(request, cancellationToken);
        if (!validation.IsValid)
        {
            return ValidationProblem(validation.ToDictionary());
        }

        var response = await _registration.RegisterAsync(request, cancellationToken);
        return Created($"/api/customers/{response.UserId}", response);
    }
}

        _auth = auth;

Flow Guides

Password Login

1Frontend posts email/password to login endpoint.
2API optionally verifies reCAPTCHA token.
3IAuthenticationService.LoginAsync validates user and lockout/email status.
4API returns tokens or a 2FA challenge.

Google/Facebook Social Login

1Frontend navigates to your /auth/google or /auth/facebook endpoint.
2Endpoint redirects to provider authorization URL with callback URI.
3Provider redirects back with code.
4ISocialLoginService exchanges code, links/registers user, returns JWT result.

Google reCAPTCHA

1Frontend loads Google script with the site key.
2Frontend executes reCAPTCHA and sends token with login/register request.
3API calls IRecaptchaService.VerifyTokenAsync(token, "login").
4API continues only when Success is true and score is acceptable.

Refresh Token Rotation

1Client stores refresh token securely after login.
2Client posts refresh token when access token expires.
3DainnUser validates and rotates the refresh token.
4Reuse of a consumed token is treated as a security incident.

Common Mistakes

Required

Installing DainnUser.Api as a package

Do not do this. DainnUser.Api is reference source only. Copy the patterns into your host app instead.

Required

Keeping secrets in source control

Move JWT, email, social login, and captcha secrets to user-secrets, env vars, or a secret manager.

Optional

Using the wrong callback URI

Social login fails if Google/Facebook/Microsoft/GitHub callback URLs do not match the provider config exactly.

Required

Skipping migrations

The default stack expects the database schema to exist before login and registration are tested.

Advanced

Editing DainnUser entities for app-specific fields

Keep business fields in your own tables linked by UserId. Do not force tenant/profile data into auth tables.

Advanced

Storing refresh tokens unsafely

Refresh tokens are sensitive. Use a secure browser storage strategy or server-side session strategy based on your app architecture.

Production Minimum

This is the smallest set of checks I would expect before shipping a DainnUser-backed user module.

1Use strong secrets for JWT, email, social providers, and reCAPTCHA. Do not commit them to git.
2Apply EF migrations in CI/CD before the app serves traffic.
3Enable HTTPS and bearer authentication, and verify protected endpoints reject missing tokens.
4Configure email delivery if registration verification, password reset, or contact verification is enabled.
5Set rate limiting on auth endpoints if the public login surface is exposed.
6Run vulnerability audit on NuGet packages before release.
If the host app owns a custom email or storage service, register it before AddDainnUser() so the default validation does not force provider config you do not use.
