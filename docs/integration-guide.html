Data Flow

1. User Registration: User signs up → DainnUser creates User record → DainnStripe creates Stripe Customer → Link User.Id ↔ Customer.Id
2. Subscription Purchase: User selects plan → DainnStripe creates Checkout Session → User completes payment → Webhook updates subscription status → DainnUser grants access
3. Access Control: User requests resource → DainnUser checks authentication → Check subscription status → Grant/deny access based on plan
4. Webhook Processing: Stripe sends event → DainnStripe verifies signature → Process event → Update database → Trigger business logic

Database Schema Integration

Both libraries use separate database contexts but can share the same database:

Library DbContext Key Tables
DainnUser DainnUserDbContext Users, Roles, Permissions, Sessions
DainnStripe DainnStripeDbContext Payments, Subscriptions, Products, Webhooks
Your App ApplicationDbContext User-Customer mappings, Business data

Step 4: Configure Program.cs

using DainnUser.Api.Extensions;
using DainnUser.Infrastructure;
using DainnStripe;

var builder = WebApplication.CreateBuilder(args);

// Add DainnUser core services
builder.Services.AddDainnUser(builder.Configuration);

// Add JWT Bearer authentication
builder.Services.AddDainnUserJwtAuthentication(builder.Configuration);

// Add DainnStripe services
builder.Services.AddDainnStripe(builder.Configuration);

// Add DainnStripe.DainnUser adapter (maps JWT user → Stripe customer automatically)

builder.Services.AddControllers();
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen();

var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI();
}

app.UseHttpsRedirection();

// DainnUser rate limiting (optional)
app.UseDainnUser();

// DainnStripe request logging (optional)
app.UseDainnStripe();

app.UseAuthentication();
app.UseAuthorization();

// Map Stripe webhooks and commerce endpoints
app.MapDainnStripeWebhooks();
app.MapDainnStripeCommerceEndpoints();

app.MapControllers();

app.Run();

💡 DainnStripe.DainnUser Adapter: The adapter bridges DainnUser's JWT identity with DainnStripe's customer system. It automatically resolves the current user's Stripe Customer, creating one on first use. Register it with AddDainnStripeDainnUserIntegration().

Step 7: Use the Adapter in Your Services

// The adapter registers IDainnUserStripeCustomerResolver for you.
// Inject it wherever you need to look up or lazily create the Stripe Customer
// for the authenticated user.

using DainnStripe.DainnUser.Interfaces;

public class BillingService
{
    private readonly IDainnUserStripeCustomerResolver _customerResolver;
    private readonly IDainnStripeCheckoutService _checkoutService;

    public BillingService(
        IDainnUserStripeCustomerResolver customerResolver,
        IDainnStripeCheckoutService checkoutService)
    {
        _customerResolver = customerResolver;
        _checkoutService = checkoutService;
    }

    public async Task<string> StartCheckoutAsync(string priceId)
    {
        // Resolves or lazily creates the Stripe Customer from the current JWT claims
        var customerId = await _customerResolver.ResolveOrCreateAsync();

        var session = await _checkoutService.CreateAsync(new CreateCheckoutSessionRequest
        {
            StripeCustomerId = customerId,
            Mode = DainnStripeCheckoutMode.Subscription,
            SuccessUrl = "https://myapp.com/billing/success?session_id={CHECKOUT_SESSION_ID}",
            CancelUrl = "https://myapp.com/pricing",
            LineItems = new() { new() { PriceId = priceId, Quantity = 1 } }
        });

        return session.Url;
    }
}
💡 StripeCustomerMapping: DainnStripe stores the Owner→Customer link in the built-in StripeCustomerMapping entity (DainnStripeDbContext). You don't need to create your own mapping table.

User-Payment Linking Patterns

Pattern 1: Built-in Adapter (Recommended)

Recommended Install DainnStripe.DainnUser and call AddDainnStripeDainnUserIntegration(). The adapter uses the built-in StripeCustomerMapping entity and IDainnUserStripeCustomerResolver to lazily create Stripe Customers from JWT claims. No extra table needed.

// Register adapter in Program.cs
builder.Services.AddDainnStripeDainnUserIntegration();

// Use resolver in your service
public class BillingService(IDainnUserStripeCustomerResolver resolver)
{
    public async Task<string> GetCustomerIdAsync()
        => await resolver.ResolveOrCreateAsync(); // creates if not found
}

Pattern 2: Direct Mapping Table (Manual)

Alternative Use the built-in StripeCustomerMapping entity directly from DainnStripeDbContext.

using DainnStripe.Entities;

// Query the built-in mapping table
var mapping = await _dainnStripeDbContext.StripeCustomerMappings
    .FirstOrDefaultAsync(m => m.OwnerId == userId);

if (mapping != null)
    return mapping.StripeCustomerId;

Pattern 3: Extended User Entity

Simple Add Stripe fields directly to User entity.

public class ApplicationUser : User
{
    public string? StripeCustomerId { get; set; }
    public string? StripeSubscriptionId { get; set; }
    public DateTime? SubscriptionExpiresAt { get; set; }
}

// Usage
var user = await _userManager.GetUserAsync(User);
var customerId = user.StripeCustomerId;

Pattern 4: Metadata-Based Linking

Stripe-First Store User ID in Stripe Customer metadata.

// Create customer with metadata
var customer = await customerService.CreateAsync(new CustomerCreateOptions
{
    Email = user.Email,
    Metadata = new Dictionary<string, string>
    {
        { "user_id", user.Id },
        { "tenant_id", tenantId }
    }
});

// Query by metadata (requires Stripe API call)
var customers = await customerService.ListAsync(new CustomerListOptions
{
    Email = email
});

Comparison

Pattern Pros Cons Best For
Adapter (DainnStripe.DainnUser) Zero boilerplate, automatic customer creation, built-in mapping table Extra package dependency Most applications
Built-in Mapping Table Clean separation, flexible, easy to query locally Need to manage customer creation yourself Custom resolution logic
Extended Entity Simple, fewer queries Couples User entity to Stripe concepts Simple single-tenant SaaS
Metadata No local storage needed Requires API calls, slower lookups Stripe-centric apps

SaaS Integration Example

Build a subscription-based SaaS application with tiered pricing and feature gating.

Scenario

A project management SaaS with three tiers:

  • Free: 3 projects, 5 team members
  • Pro ($29/month): Unlimited projects, 20 team members
  • Enterprise ($99/month): Everything + priority support

Step 1: Define Subscription Plans

// Models/SubscriptionPlan.cs
public enum SubscriptionTier
{
    Free,
    Pro,
    Enterprise
}

public class SubscriptionPlan
{
    public SubscriptionTier Tier { get; set; }
    public string StripePriceId { get; set; } = string.Empty;
    public int MaxProjects { get; set; }
    public int MaxTeamMembers { get; set; }
    public bool HasPrioritySupport { get; set; }
}

// Configuration
public static class Plans
{
    public static readonly SubscriptionPlan Free = new()
    {
        Tier = SubscriptionTier.Free,
        StripePriceId = "",
        MaxProjects = 3,
        MaxTeamMembers = 5,
        HasPrioritySupport = false
    };

    public static readonly SubscriptionPlan Pro = new()
    {
        Tier = SubscriptionTier.Pro,
        StripePriceId = "price_pro_monthly",
        MaxProjects = int.MaxValue,
        MaxTeamMembers = 20,
        HasPrioritySupport = false
    };

    public static readonly SubscriptionPlan Enterprise = new()
    {
        Tier = SubscriptionTier.Enterprise,
        StripePriceId = "price_enterprise_monthly",
        MaxProjects = int.MaxValue,
        MaxTeamMembers = int.MaxValue,
        HasPrioritySupport = true
    };
}

Step 2: Create Subscription Service

public interface ISubscriptionService
{
    Task<string> CreateCheckoutSessionAsync(string userId, SubscriptionTier tier);
    Task<SubscriptionTier> GetUserSubscriptionTierAsync(string userId);
    Task<bool> CanUserAccessFeatureAsync(string userId, string feature);
}

public class SubscriptionService : ISubscriptionService
{
    private readonly IUserPaymentService _userPaymentService;
    private readonly IDainnStripeCheckoutService _checkoutService;
    private readonly ApplicationDbContext _dbContext;

    public async Task<string> CreateCheckoutSessionAsync(string userId, SubscriptionTier tier)
    {
        var user = await _dbContext.Users.FindAsync(userId);
        var customerId = await _userPaymentService.GetOrCreateStripeCustomerAsync(userId, user.Email);

        var plan = tier switch
        {
            SubscriptionTier.Pro => Plans.Pro,
            SubscriptionTier.Enterprise => Plans.Enterprise,
            _ => throw new ArgumentException("Invalid tier")
        };

        var request = new CreateCheckoutSessionRequest
        {
            CustomerId = customerId,
            Mode = DainnStripeCheckoutMode.Subscription,
            SuccessUrl = "https://myapp.com/subscription/success?session_id={CHECKOUT_SESSION_ID}",
            CancelUrl = "https://myapp.com/pricing",
            LineItems = new List<CreateCheckoutSessionLineItem>
            {
                new() { PriceId = plan.StripePriceId, Quantity = 1 }
            }
        };

        var result = await _checkoutService.CreateAsync(request);
        return result.Url;
    }

    public async Task<SubscriptionTier> GetUserSubscriptionTierAsync(string userId)
    {
        var subscription = await _dbContext.UserSubscriptions
            .Where(s => s.UserId == userId && s.Status == "active")
            .OrderByDescending(s => s.CreatedAt)
            .FirstOrDefaultAsync();

        return subscription?.Tier ?? SubscriptionTier.Free;
    }

    public async Task<bool> CanUserAccessFeatureAsync(string userId, string feature)
    {
        var tier = await GetUserSubscriptionTierAsync(userId);
        var plan = GetPlanForTier(tier);

        return feature switch
        {
            "unlimited_projects" => plan.MaxProjects == int.MaxValue,
            "priority_support" => plan.HasPrioritySupport,
            _ => true
        };
    }
}

Step 3: Feature Gating with Authorization

// Authorization/SubscriptionRequirement.cs
public class SubscriptionRequirement : IAuthorizationRequirement
{
    public SubscriptionTier MinimumTier { get; }

    public SubscriptionRequirement(SubscriptionTier minimumTier)
    {
        MinimumTier = minimumTier;
    }
}

public class SubscriptionHandler : AuthorizationHandler<SubscriptionRequirement>
{
    private readonly ISubscriptionService _subscriptionService;

    public SubscriptionHandler(ISubscriptionService subscriptionService)
    {
        _subscriptionService = subscriptionService;
    }

    protected override async Task HandleRequirementAsync(
        AuthorizationHandlerContext context,
        SubscriptionRequirement requirement)
    {
        var userId = context.User.FindFirst(ClaimTypes.NameIdentifier)?.Value;
        if (userId == null)
        {
            context.Fail();
            return;
        }

        var userTier = await _subscriptionService.GetUserSubscriptionTierAsync(userId);
        if (userTier >= requirement.MinimumTier)
        {
            context.Succeed(requirement);
        }
    }
}

// Register in Program.cs
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("RequiresPro", policy =>
        policy.Requirements.Add(new SubscriptionRequirement(SubscriptionTier.Pro)));

    options.AddPolicy("RequiresEnterprise", policy =>
        policy.Requirements.Add(new SubscriptionRequirement(SubscriptionTier.Enterprise)));
});

builder.Services.AddScoped<IAuthorizationHandler, SubscriptionHandler>();

Step 5: Handle Subscription Webhooks

public class SubscriptionWebhookHandler : IStripeWebhookHandler
{
    private readonly ApplicationDbContext _dbContext;

    public bool CanHandle(string eventType)
        => eventType.StartsWith("customer.subscription.");

    public async Task HandleAsync(
        Event stripeEvent,
        StripeWebhookEventRecord record,
        CancellationToken cancellationToken = default)
    {
        var subscription = stripeEvent.Data.Object as Subscription;
        var userId = subscription.Metadata["user_id"];

        switch (stripeEvent.Type)
        {
            case "customer.subscription.created":
            case "customer.subscription.updated":
                await UpdateUserSubscriptionAsync(userId, subscription);
                break;

            case "customer.subscription.deleted":
                await CancelUserSubscriptionAsync(userId);
                break;
        }
    }

    private async Task UpdateUserSubscriptionAsync(string userId, Subscription subscription)
    {
        var userSub = await _dbContext.UserSubscriptions
            .FirstOrDefaultAsync(s => s.UserId == userId);

        if (userSub == null)
        {
            userSub = new UserSubscription { UserId = userId };
            _dbContext.UserSubscriptions.Add(userSub);
        }

        userSub.StripeSubscriptionId = subscription.Id;
        userSub.Status = subscription.Status;
        userSub.CurrentPeriodEnd = subscription.CurrentPeriodEnd;
        userSub.Tier = DetermineTierFromPriceId(subscription.Items.Data[0].Price.Id);

        await _dbContext.SaveChangesAsync();
    }
}

Marketplace Integration Example

Build a multi-tenant marketplace platform with seller onboarding and split payments using Stripe Connect.

Scenario

An online marketplace where:

  • Sellers create accounts and list products
  • Buyers purchase from sellers
  • Platform takes 10% commission
  • Sellers receive payouts to their bank accounts

Step 1: Extend User Model for Sellers

public class ApplicationUser : User
{
    public UserType Type { get; set; } = UserType.Buyer;
    public string? StripeCustomerId { get; set; }
    public string? StripeConnectedAccountId { get; set; }
    public bool IsSellerOnboarded { get; set; }
}

public enum UserType
{
    Buyer,
    Seller
}

Step 2: Seller Onboarding Service

public interface ISellerOnboardingService
{
    Task<string> CreateSellerAccountAsync(string userId);
    Task<string> CreateOnboardingLinkAsync(string userId);
    Task<bool> IsSellerOnboardedAsync(string userId);
}

public class SellerOnboardingService : ISellerOnboardingService
{
    private readonly IDainnStripeConnectService _connectService;
    private readonly ApplicationDbContext _dbContext;

    public async Task<string> CreateSellerAccountAsync(string userId)
    {
        var user = await _dbContext.Users.FindAsync(userId);

        // Create Stripe Connected Account
        var request = new CreateConnectedAccountRequest
        {
            Type = "express", // or "standard" for more control
            Email = user.Email,
            Metadata = new Dictionary<string, string>
            {
                { "user_id", userId }
            }
        };

        var result = await _connectService.CreateConnectedAccountAsync(request);

        // Save to user
        user.StripeConnectedAccountId = result.AccountId;
        await _dbContext.SaveChangesAsync();

        return result.AccountId;
    }

    public async Task<string> CreateOnboardingLinkAsync(string userId)
    {
        var user = await _dbContext.Users.FindAsync(userId);

        if (string.IsNullOrEmpty(user.StripeConnectedAccountId))
        {
            await CreateSellerAccountAsync(userId);
            user = await _dbContext.Users.FindAsync(userId);
        }

        var linkRequest = new CreateConnectedAccountLinkRequest
        {
            AccountId = user.StripeConnectedAccountId,
            RefreshUrl = "https://mymarketplace.com/seller/onboarding/refresh",
            ReturnUrl = "https://mymarketplace.com/seller/dashboard",
            Type = "account_onboarding"
        };

        var result = await _connectService.CreateOnboardingLinkAsync(linkRequest);
        return result.Url;
    }

    public async Task<bool> IsSellerOnboardedAsync(string userId)
    {
        var user = await _dbContext.Users.FindAsync(userId);
        return user.IsSellerOnboarded;
    }
}

Step 3: Split Payment Processing

public class MarketplacePaymentService
{
    private readonly IDainnStripePaymentService _paymentService;
    private readonly ApplicationDbContext _dbContext;
    private const decimal PLATFORM_FEE_PERCENTAGE = 0.10m; // 10%

    public async Task<PaymentIntentResult> CreateMarketplacePaymentAsync(
        string buyerId,
        string sellerId,
        decimal amount,
        string orderId)
    {
        var seller = await _dbContext.Users.FindAsync(sellerId);
        var buyer = await _dbContext.Users.FindAsync(buyerId);

        if (string.IsNullOrEmpty(seller.StripeConnectedAccountId))
            throw new InvalidOperationException("Seller not onboarded");

        // Calculate platform fee
        var platformFee = (long)(amount * PLATFORM_FEE_PERCENTAGE * 100); // in cents

        var request = new CreatePaymentIntentRequest
        {
            Amount = (long)(amount * 100), // Convert to cents
            Currency = "usd",
            CustomerId = buyer.StripeCustomerId,
            ApplicationFeeAmount = platformFee,
            TransferData = new PaymentIntentTransferDataOptions
            {
                Destination = seller.StripeConnectedAccountId
            },
            Metadata = new Dictionary<string, string>
            {
                { "order_id", orderId },
                { "buyer_id", buyerId },
                { "seller_id", sellerId }
            }
        };

        return await _paymentService.CreatePaymentIntentAsync(request);
    }
}

Step 5: Handle Connect Webhooks

public class ConnectWebhookHandler : IStripeWebhookHandler
{
    private readonly ApplicationDbContext _dbContext;

    public bool CanHandle(string eventType)
        => eventType.StartsWith("account.");

    public async Task HandleAsync(
        Event stripeEvent,
        StripeWebhookEventRecord record,
        CancellationToken cancellationToken = default)
    {
        var account = stripeEvent.Data.Object as Account;
        var userId = account.Metadata["user_id"];

        switch (stripeEvent.Type)
        {
            case "account.updated":
                await UpdateSellerAccountStatusAsync(userId, account);
                break;

            case "account.external_account.created":
                // Seller added bank account
                await NotifySellerBankAccountAddedAsync(userId);
                break;
        }
    }

    private async Task UpdateSellerAccountStatusAsync(string userId, Account account)
    {
        var user = await _dbContext.Users.FindAsync(userId);
        user.IsSellerOnboarded = account.ChargesEnabled && account.PayoutsEnabled;
        await _dbContext.SaveChangesAsync();
    }
}

Architecture Overview

Marketplace Flow:

1. Seller Onboarding:
   User registers → DainnUser creates User
   → User becomes Seller → DainnStripe creates Connected Account
   → Stripe onboarding → Seller verified

2. Product Listing:
   Seller creates product → Store in database
   → Link to Seller's Connected Account

3. Purchase Flow:
   Buyer selects product → Create Payment Intent
   → Split payment: 90% to Seller, 10% to Platform
   → Webhook confirms payment → Update order status

4. Payout:
   Stripe automatically pays out to Seller's bank account
   → Platform keeps commission

FAQ

General Questions

Q: Can I use DainnUser without DainnStripe?

A: Yes! Both libraries are independent. Use DainnUser for authentication/authorization and integrate with any payment provider you prefer.

Q: Can I use DainnStripe without DainnUser?

A: Yes! DainnStripe works with any authentication system. You'll need to manage User-Customer mappings yourself.

Q: Do both libraries share the same database?

A: They can, but use separate DbContexts. Each library manages its own tables. You create the linking tables in your application's DbContext.

Integration Questions

Q: How do I link a User to a Stripe Customer?

A: Four patterns, from simplest to most custom:

  1. Adapter (Recommended): Install DainnStripe.DainnUser, call AddDainnStripeDainnUserIntegration(), inject IDainnUserStripeCustomerResolver
  2. Built-in Mapping Table: Query DainnStripeDbContext.StripeCustomerMappings (already exists, no extra table needed)
  3. Extended Entity: Add StripeCustomerId to your User entity
  4. Metadata: Store owner_id in Stripe Customer metadata

Q: When should I create the Stripe Customer?

A: Two approaches:

  • On Registration: Create immediately when user signs up (simpler)
  • On First Payment: Create when user makes first payment (lazy loading)

Q: How do I handle subscription-based access control?

A: Use custom authorization policies that check subscription status. See the SaaS example in this guide.

Webhook Questions

Q: What happens if a webhook fails?

A: Stripe automatically retries failed webhooks with exponential backoff. DainnStripe tracks processing status in the database.

Q: How do I test webhooks locally?

A: Use Stripe CLI: stripe listen --forward-to https://localhost:5001/stripe/webhooks

Q: Can I process the same webhook event twice?

A: No. DainnStripe automatically handles idempotency using the StripeWebhookEventRecords table.

Payment Questions

Q: Should I use Payment Intents or Checkout Sessions?

A:

  • Payment Intents: For custom payment flows with your own UI
  • Checkout Sessions: For hosted Stripe payment pages (easier, faster)

Q: How do I handle failed payments?

A: Listen to payment_intent.payment_failed webhook. Notify user and optionally retry or downgrade subscription.

Q: Can I offer free trials?

A: Yes! Set TrialPeriodDays when creating subscription. Stripe won't charge until trial ends.

Marketplace Questions

Q: What's the difference between Express and Standard Connect accounts?

A:

  • Express: Stripe handles onboarding UI (easier, recommended)
  • Standard: You build custom onboarding (more control)

Q: How do I handle platform fees?

A: Use ApplicationFeeAmount when creating Payment Intent. Platform keeps the fee, seller receives the rest.

Q: Can sellers see buyer information?

A: Depends on your implementation. With Express accounts, sellers see limited buyer info. Control this via your application logic.

Best Practices Summary

✅ Do:
  • Always verify webhook signatures
  • Use HTTPS in production
  • Store secrets securely (Key Vault, Secrets Manager)
  • Implement proper error handling and logging
  • Test thoroughly with Stripe test mode
  • Monitor webhook processing and payment success rates
  • Keep libraries updated to latest versions
⚠️ Don't:
  • Commit API keys to source control
  • Skip webhook signature verification
  • Use test keys in production
  • Trust client-side payment amounts
  • Ignore failed webhook events
  • Deploy without testing in staging first
